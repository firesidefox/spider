# System Prompt Layered Architecture Design

## Goal

Refactor system prompt construction from a centralized string in `factory.go` into a three-layer architecture where each tool owns its behavior guidance.

## Current State

All tool behavior rules live in two large string constants in `factory.go`:
- `toolBehaviorPrompt` — rules for all tools (RunCommand, VerifyTool, CallAPI, etc.)
- `todoTaskPrompt` — rules for TodoTaskTool
- `complexTaskPrompt` — cross-tool orchestration rules

Adding a new tool requires editing both the tool file and `factory.go`. Rules and implementation are decoupled.

## Three-Layer Architecture

### Layer 1: Environment (dynamic)

Generated by `BuildSystemPrompt()`. Contains role definition and host inventory.

```
You are Spider, an intelligent network operations assistant.
You manage N network devices: vendor(count)...
```

### Layer 2: Tool Layer (static, per-tool)

Each tool that has behavior guidance implements the `SystemPromptSection` interface:

```go
type SystemPromptSection interface {
    SystemPromptSection() string
}
```

`BuildSystemPrompt` collects sections from all registered tools in registration order and appends them after Layer 1.

Tools implementing `SystemPromptSection`:
- `ExecuteCLITool` — RunCommand/RunCommandBatch rules
- `BatchExecuteTool` — same content as ExecuteCLITool (shared via package-level constant)
- `VerifyTool` — VerifyTool when-to-use / when-not-to-use
- `CallRESTAPITool` — CallAPI GET vs POST/PUT/DELETE rules
- `TodoTaskTool` — Task Management rules (moved from `todoTaskPrompt`)
- `ListDevicesTool`, `GetDeviceInfoTool`, `SearchDocsTool` — single combined section via `ListDevicesTool`

Tools not implementing `SystemPromptSection` contribute nothing. No change required.

### Layer 3: Orchestration (static, factory-level)

Two constants remain in `factory.go` — cross-tool rules that belong to no single tool:

- `intentFieldPrompt` — Intent Field rules (applies to RunCommand, BatchExecute, CallAPI)
- `orchestrationPrompt` — Complex Multi-Step Tasks (renamed from `complexTaskPrompt`)

## Assembly Order

```
Layer 1 (dynamic host summary)
+ Layer 2 (tool sections, registration order)
+ Layer 3 (intentFieldPrompt + orchestrationPrompt)
```

`BuildSystemPrompt` assembles Layer 2 with a `strings.Builder`. Empty sections are skipped — a tool returning an empty or whitespace-only string from `SystemPromptSection()` contributes nothing, and no separator is inserted:

```go
var b strings.Builder
b.WriteString(layer1)
for _, tool := range registry.All() {
    if sp, ok := tool.(SystemPromptSection); ok {
        section := sp.SystemPromptSection()
        if strings.TrimSpace(section) != "" {
            b.WriteString("\n\n")
            b.WriteString(section)
        }
    }
}
b.WriteString("\n\n")
b.WriteString(intentFieldPrompt)
b.WriteString("\n\n")
b.WriteString(orchestrationPrompt)
return b.String()
```

## ToolRegistry Changes

Current: `map[string]Tool` — unordered.

Change to ordered registry:

```go
type ToolRegistry struct {
    tools []Tool
    index map[string]int
}
```

New method:

```go
func (r *ToolRegistry) All() []Tool
```

`Register`, `Get`, `Definitions` behavior unchanged. Registration order determines Layer 2 prompt order.

## Shared Prompt Content

`ExecuteCLITool` and `BatchExecuteTool` cover the same RunCommand/RunCommandBatch section. Extract to a package-level constant `runCommandPromptSection` in `tools_cli.go`. Both tools return it from `SystemPromptSection()`.

`ListDevicesTool`, `GetDeviceInfoTool`, and `SearchDocsTool` share one section ("read-only, call freely"). Only `ListDevicesTool` implements `SystemPromptSection` — it returns the combined section for all three. The other two do not implement the interface.

## Files Changed

| File | Change |
|------|--------|
| `internal/agent/tools.go` | Add `SystemPromptSection` interface, refactor `ToolRegistry` to ordered |
| `internal/agent/factory.go` | `BuildSystemPrompt` collects Layer 2 from registry; rename `complexTaskPrompt` → `orchestrationPrompt`; add `intentFieldPrompt`; remove `toolBehaviorPrompt` and `todoTaskPrompt` |
| `internal/agent/tools_cli.go` | Add `SystemPromptSection()` to `ExecuteCLITool`; extract `runCommandPromptSection` constant |
| `internal/agent/tools_batch.go` | Add `SystemPromptSection()` to `BatchExecuteTool` returning `runCommandPromptSection` |
| `internal/agent/tools_verify.go` | Add `SystemPromptSection()` to `VerifyTool` |
| `internal/agent/tools_api.go` | Add `SystemPromptSection()` to `CallRESTAPITool` |
| `internal/agent/tools_todo_task.go` | Add `SystemPromptSection()` to `TodoTaskTool` |
| `internal/agent/tools_list_devices.go` | Add `SystemPromptSection()` to `ListDevicesTool` (covers all three read-only tools) |

## What Does Not Change

- `Description()` on all tools — stays as one-line API contract
- Tool registration order in `NewAgent` — unchanged, determines Layer 2 order
- Layer 3 content — same rules, just renamed and split
- No layer boundary markers — all three layers are static, no cache split needed
